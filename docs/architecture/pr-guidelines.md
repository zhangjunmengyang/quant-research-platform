# PR 规范（必须遵守）

本文档定义本仓库的 Pull Request 最低要求。所有 PR 在创建前、提交前、请求评审前都必须对照本规范自检。

不满足本规范的 PR，默认结论是 `Request changes` 或直接关闭，不进入“边改边审”模式。

## 1. 核心原则

1. 单一目标。一个 PR 只能有一个主目标，必须能用一句话说清楚。
2. 可独立合并。PR 合入后，主分支必须保持可运行、可测试、可继续开发。
3. 架构优先。不得为了“先做出来”牺牲分层、边界、接线完整性和文档准确性。
4. 小步提交。默认优先拆分为多个小 PR，而不是提交一个大而杂的 PR。
5. 公私分离。私有资产、专有框架、本机路径、本机脚本不得混入主仓库基线。

## 2. 硬性要求

### 2.1 分支约束

凡是走 PR，必须满足以下要求：

- 必须从新的功能分支或修复分支发起 PR。
- 禁止直接在 `main` 或其他长期分支上开发后再补 PR。
- 禁止直接 `push` 到主分支，再把该提交当作“等价于已评审 PR”。
- 一个分支只服务一个主目标，不得长期堆积多个独立需求。

推荐命名：

- `feat/<topic>`
- `fix/<topic>`
- `refactor/<topic>`
- `docs/<topic>`

### 2.2 范围约束

PR 必须满足以下要求：

- 只解决一个主问题，不得混入多个彼此独立的主题。
- 不得把“新增业务域 + 前端页面 + MCP 服务 + 部署编排 + 数据资产迁移 + 临时脚本”混在一个 PR 中。
- 不得同时混入功能开发、重构、目录搬迁、数据补录、历史垃圾清理，除非这些变更严格服务于同一条交付链路，且缺一不可。

### 2.3 大小约束

默认要求：

- 单个 PR 变更文件数不超过 `20` 个。
- 单个 PR 有效代码改动不超过 `800` 行。

超过以上阈值时：

- 必须拆分。
- 必须在 PR 描述中解释为什么不能拆。

满足以下任一条件时，默认直接拒绝：

- 跨越两个及以上独立主题。
- 出现大批与主目标无关的新增文件。
- 评审者无法在一次审查中确认架构、安全、功能闭环。

说明：

- 生成文件、锁文件、纯文档可酌情单独评估，但不能作为掩盖大 PR 的理由。

### 2.4 接线完整性

任何新增能力都必须在同一 PR 内完成主系统接线，禁止提交死代码：

- 新增 REST 路由，必须接入总路由。
- 新增前端页面，必须接入主路由、导航和必要的标题/i18n。
- 新增 MCP 服务，必须接入管理面、端口配置、启动编排和文档。
- 新增任务链路，必须说明入口、触发方式、状态存储和失败处理。

未接线但“未来会接”的代码，不得进入主分支。

### 2.5 文档同步

变更触发以下场景时，必须同步更新文档：

- 新增/删除业务域。
- 新增/修改 API 模式。
- 新增/修改 MCP 工具或服务。
- 修改服务端口。
- 修改目录结构。
- 新增开发约定、提交流程或评审规则。

至少要更新：

- `AGENTS.md`
- `docs/architecture/reference.md`
- 对应任务文档
- 本文档（如果 PR 规范本身发生变化）

### 2.6 测试与验证

PR 必须提供可复现的验证方式：

- 必须列出执行过的命令。
- 必须说明结果是通过、失败还是未执行。
- 测试必须能在仓库标准环境中运行，不能依赖作者本机私有路径、私有框架、手工数据目录或本机 GUI。
- 仅能在作者电脑上运行的“脚本式测试”，不算有效测试覆盖。

禁止：

- 把测试放在默认不会被 CI/pytest 发现的位置，却声称“已经有测试”。
- 用打印输出代替断言。
- 用真实生产级长任务代替可重复的单元/集成测试。

### 2.7 安全与运维约束

以下内容默认禁止进入主仓：

- 对不受信任输入执行 `pickle.load` 或其他危险反序列化。
- 将长时任务直接挂在 HTTP 请求链路上阻塞 API worker。
- 通过命令行参数、日志、响应体暴露密钥、Token、本机绝对路径、专有框架路径。
- 将私有研究资产、专有代码、数据样本直接提交到主仓或打进镜像。
- 用手写 HTTP 逻辑重复实现已有基础设施能力，例如绕过 `mcp_core/llm` 直接手写 LLM 调用。

### 2.8 仓库边界

以下内容必须留在私有区或外部依赖中，不得直接提交到主仓：

- `private/` 约定下的私有因子、截面因子、元数据。
- 专有研究框架。
- 作者本机 PowerShell/批处理启动脚本，除非它们已被仓库正式采纳并文档化。
- 仅服务于个人实验的临时目录、缓存、样本结果、分析产物。

## 3. PR 描述模板要求

每个 PR 描述至少必须包含以下内容：

1. 背景与目标
2. 本 PR 范围
3. 明确不包含什么
4. 架构影响
5. 文档更新清单
6. 验证命令与结果
7. 风险点
8. 回滚方式

缺失以上任一项，评审者有权直接退回。

## 4. 推荐拆分方式

### 4.1 新增业务域

按以下顺序拆分：

1. 业务域脚手架与架构文档 PR
2. 后端接口与 schema PR
3. 前端接线 PR
4. MCP/运维编排 PR
5. 测试与补充文档 PR

### 4.2 大功能改造

按以下顺序拆分：

1. 纯重构 PR
2. 协议/接口变更 PR
3. 功能落地 PR
4. 清理与文档 PR

### 4.3 数据与资产迁移

按以下顺序拆分：

1. 目录和边界约束 PR
2. 工具链/同步机制 PR
3. 资产迁移 PR

禁止把“规则建立”和“资产大规模导入”放在同一个 PR。

## 5. 默认拒绝场景

出现以下任一情况时，评审默认给出拒绝或要求大幅拆分：

- 主功能没有接入系统入口。
- 代码不可达或没有用户入口。
- 文档基线未更新。
- 测试不可复现。
- 混入大量私有资产或作者本机依赖。
- 出现明显安全/运维风险。
- PR 大到无法一次性完成高质量评审。

## 6. 评审结论标准

- `Approve`：范围清晰、接线完整、测试可复现、文档同步、安全可接受。
- `Request changes`：存在明确缺陷，但修复后仍适合继续在当前 PR 上推进。
- `Close and split`：范围失控、主题混杂、架构污染严重，继续在当前 PR 上修补没有意义。

## 7. 给 Claude Code / Codex 的使用要求

如果使用 Claude Code、Codex 或类似代理协助开发，提交前必须先让代理完成以下自检：

1. 检查是否存在未接线死代码
2. 检查是否违反架构文档
3. 检查是否缺失文档更新
4. 检查测试是否可被默认命令发现并运行
5. 检查是否混入私有资产、本机依赖和安全风险
6. 输出拆分建议，确认当前 PR 是否应拆成多个 PR

代理输出不能替代人工判断，但必须作为提交前的最低质量门槛。
